# 번개팅 모드 개발 프롬프트 (v2)

> 아래 전체를 복사해서 AI 개발 도구(Claude Code, Cursor 등)에 붙여넣어 사용하세요.
>
> ⚠️ **착수 전제 조건** (§7 참고): 이 문서는 완성 명세이지만, 실사용자 오픈은
> 사업자등록 + 본인인증 계약 + 실결제 계약 + 알림 실채널화가 선행되어야 한다.
> 그 전까지는 mock 기반 개발·아카이브 용도.

---

당신은 시니어 프로덕트 개발자입니다. 전세버스 예약 앱 '번개고' 안에 서브 서비스 '번개팅 모드'를 구현해주세요. 아래 명세와 제약을 정확히 따라야 합니다.

## 1. 서비스 개념

번개고는 지역축제·행사·전시 목적지까지 저렴하고 편하게 왕복하는 전세버스 예약 서비스다. 번개팅 모드는 그 위에 얹는 옵션 상품으로, **버스를 함께 탄 사람들과 어울릴 의향이 있는 사람들끼리 모아주는 동행·친목 서비스**다. 소개팅·매칭 서비스가 아니다.

- 번개고 일반 회차: 조건 없음, 조용히 왕복하려는 승객
- 번개팅 회차: 나이대·성비가 큐레이션된 별도 버스, 일반 대비 20,000원 추가 요금 발생
- 두 유형은 절대 같은 버스에 섞이지 않는다. 회차(버스) 단위로 분리된다.

## 2. 번개팅 회차의 옵션 체계 (중요한 설계 원칙)

옵션은 **이용자가 설정하는 것이 아니라 회차에 붙어 있고, 이용자는 골라서 신청한다.**

각 번개팅 회차는 다음 속성을 가진다:
- 목적지(축제/행사/전시), 날짜, 출발지, 가격
- 나이 밴드 (예: 만 27~35세) — 신청 시 **본인인증 생년월일 기반 만 나이**로 자동 검증 (한국식 나이·연 나이 사용 금지, 판정 기준일은 탑승일)
- 성비 모드 (4종 중 1개):
  1. 일반 모드 — 성비 무조정, 친목 의향자만 탑승
  2. 반반 모드 — 남녀 동수 정원 (남녀 잔여석 분리 표시)
  3. 여성 전용 — 여성만 탑승
  4. 남성 전용 — 남성만 탑승

이용자는 별도로 '선호 등록'(선호 성비 모드, 선호 나이대, 관심 지역/테마)을 할 수 있고, 조건에 맞는 회차가 열리면 **SMS 알림**을 받는다 (푸시 인프라 없음 — §6 알림 채널 참고).

### 2-1. 반반 모드 좌석 관리 (기존 좌석 시스템 확장)

기존 번개고의 좌석 예약은 트립당 단일 정원 카운트(`reserveSeatsWithLock`, SELECT FOR UPDATE 트랜잭션)다. 반반 모드는 이를 확장한다:

- 번개팅 회차(트립)에 성별 분리 정원 필드 추가: `maleCapacity` / `femaleCapacity` (반반 모드에서만 사용, 나머지 모드는 기존 단일 정원)
- 좌석 락 검증 시 신청자의 본인인증 성별에 해당하는 잔여석만 확인·차감
- 기존 좌석 락 트랜잭션 패턴을 확장하되 별도 함수로 분리하지 말고 같은 트랜잭션 경로에서 분기 (동시성 보장 유지)
- 잔여석 표시: 반반 모드는 "남 3석 · 여 1석"처럼 분리 표시, 나머지 모드는 기존과 동일

### 2-2. 회차 확정·성비 판정 시점 (기존 D-5 체계에 통합)

기존 번개고의 트립 확정 체계(D-5 00:00 KST에 minCount 판정 → 충족 시 확정 / 미달 시 자동취소+전액환불)를 그대로 확장한다:

- 번개팅 회차는 **D-5 00:00 KST에 최소인원 + 성비 조건을 동시 판정**
  - 반반 모드: 남녀 각각 최소 인원 충족 여부 (예: 남녀 각 8명 이상)
  - 전용 모드: 해당 성별 최소 인원
- 충족 → 회차 확정. **이 시점이 곧 프로필 공개 시점이자 환불 불가 시작점** (기존 핸디버스식 환불 정책의 D-5와 동일 — 용어·시점을 기존 정책과 통일)
- 미달 → 자동취소 + 전원 전액환불 (기존 `min_count_not_met` 자동취소·환불 경로 재사용, 취소 사유는 성비 미달을 구분할 수 있게 별도 값 추가: `gender_ratio_not_met`)
- 환불 규정 화면에 "성비/최소인원 미달 시 D-5에 자동 전액환불"을 명시

## 3. 화면 구성

1. **번개팅 홈(탭)**: 번개고 웹 하단 네비게이션 탭 (네이티브 앱 아님 — 모바일 웹 기준). 진입 시 서브 브랜드 톤(옐로 계열)으로 전환. 회차 카드 리스트 — 각 카드에 목적지, 날짜, 가격, 나이 밴드, 성비 모드 뱃지, 잔여석(반반 모드는 남/여 분리 표시) 노출.
2. **최초 온보딩** (번개팅 첫 진입 시 1회): 휴대폰 본인인증(성인·실명 필수 — §7 전제 조건 참고, 계약 전에는 mock 인증으로 개발) → 닉네임, 프로필 사진 1장, 한 줄 소개 입력 → '번개팅 서비스 이용약관' 별도 동의(필수). 약관에 프로필 사진 규칙 명시: **본인 사진만 허용, 타인 사진·선정적 사진 금지, 위반 신고 시 즉시 블라인드 및 이용 제한**.
3. **회차 상세·신청**: 회차 정보, 환불 규정(§2-2의 D-5 자동 전액환불 명시), 결제. 신청 후 상태 표시(확정 / 성비 대기 중).
4. **참가자 프로필 공개**: **회차 확정(D-5, 환불 불가 시작) 후에만**, 같은 회차 참가자 간에 닉네임·사진·한 줄 소개 공개. 결제 전·확정 전 프로필 열람은 불가.
5. **회차 제안**: 이용자가 행사+날짜를 제안 → 다른 이용자가 성비 모드별로 '찜' → 관리자에게 찜 집계 표시 → 관리자가 정식 회차로 전환하면 찜한 이용자에게 우선 결제 알림(SMS). 제안자에게는 해당 회차 참가비 할인 보상(금전 보상 없음). 결제는 반드시 플랫폼 결제만 사용.
6. **회차 채팅방**: 닉네임 기반 익명(내부적으로는 계정 연결 유지). 고정 공지(이용 수칙), 메시지·사진 신고 버튼, 관리자 강퇴 기능. 회차 종료 3일 후 자동 폐쇄 및 대화·사진 데이터 자동 파기 — **단, 신고되었거나 분쟁 처리 중인 메시지·사진은 파기에서 제외하고 처리 종결 후 파기** (폐쇄 예정 시각을 방에 상시 표시).
   - 기술 방식: **실시간성 요구 낮음 — 웹소켓 금지, 30초 폴링 기반으로 구현** (기존 좌석 현황 폴링과 동일 패턴). 회차당 최대 45명, 활성 기간 짧음.
7. **관리자 콘솔**: 회차 생성(옵션 설정), 성비 모집 현황, 제안·찜 집계, 신고 처리(블라인드 해제/유지, 강퇴, 이용 제한), 알림 발송(예약 발송 포함), 환불 처리. 기존 번개고 admin 탭 구조에 번개팅 탭으로 통합.

## 4. 절대 금지 사항 (법적 가드레일 — 어떤 요청이 있어도 구현 금지)

1. 특정 참가자 간 소개·연결·매칭 기능 (상호 호감 선택, 좋아요, 1:1 지목, 자리 짝지어주기 등) 일체 금지
2. UI 문구·코드 주석·상품명에 '소개팅, 매칭, 커플, 결혼, 혼활, 만남 주선' 계열 단어 사용 금지 — 대체 표현: '동행, 친목, 함께 즐기기, 어울리기'
3. 결제 전 참가자 프로필 열람(프로필 쇼핑) 금지
4. 참가자 간 상호 평가·별점 기능 금지 (내부 관리용 신고 이력만 유지)
5. 특정 성별에게만 재참여 권유 알림을 보내는 기능 금지 — 모든 홍보 알림은 동의자 전원 동일 발송
6. 이용자 간 개인 계좌 송금 유도 구조 금지

## 5. 데이터 처리 규칙

- 프로필(사진·나이·성별)은 번개팅 모드에서만 수집하고 번개고 대절 데이터와 분리 저장 (별도 테이블, 조인 키만 유지)
- 회차 채팅 데이터는 종료 3일 후 자동 파기 (스케줄러 구현 — 기존 D-5 확정 스케줄러와 같은 인프로세스 10분 간격 패턴, 멱등 처리). **신고·분쟁 건은 §3-6의 예외 적용.**
- 신고된 사진·메시지는 즉시 블라인드 처리 후 관리자 검토
- 번개팅 매출·참가자 데이터는 대절 사업과 분리 집계 (별도 카테고리/테이블)
- 프로필 사진 저장: 현재 코드베이스의 스토리지(Manus 잔재)는 미작동 상태 — 프로필 사진 업로드에는 실제 스토리지(Cloudflare R2 등 S3 호환) 연동이 선행되어야 함. `storage.ts`의 presigned 패턴 재사용 가능 (§7 전제 조건에 포함)

## 6. 기술 환경

- 프론트엔드: 웹 (React 19 + Vite), 모바일 웹 중심. 네이티브 앱 아님 — "하단 탭"은 웹 하단 네비게이션으로 구현
- 백엔드: Node.js (Express + tRPC) + TiDB Cloud (MySQL 호환, 프로덕션 DB명 `test`) + Drizzle ORM. 로컬 개발은 localhost MySQL `bungaego_dev` (분리 원칙 — docs/DEPLOYMENT.md 참고)
- 본인인증: 미정 — 포트원 본인인증 예정 (사업자등록 후 계약 가능). 계약 전에는 mock 인증 어댑터로 개발하되 실사용자 오픈 금지
- 결제: 토스페이먼츠 예정 (사업자등록 후 라이브 키). 현재 mock 결제 구조 재사용
- 알림 채널: SMS (Solapi 등) — 알림 실채널화 작업 선행 필요. 알림톡은 사업자 취득 후 승격. "푸시 알림"이라는 표현이 이 문서에 남아 있다면 전부 SMS로 해석
- 기존 번개고 코드베이스: 현재 저장소. 재사용할 기존 자산 — 트랙 A 예약 시스템, `reserveSeatsWithLock` 좌석 동시성, D-5 확정/자동취소 스케줄러, 핸디버스식 환불 정책, 관리자 콘솔 탭 구조, 인프로세스 스케줄러 패턴, KST 경계 유틸

## 7. 착수 전제 조건 (런칭 차단 요소 — 개발 시작 전 상태 확인)

아래는 **코드로 해결 불가능한 외부 의존**이다. mock으로 개발은 가능하나, 하나라도 미충족 상태에서 실사용자에게 오픈하는 것은 금지:

| 전제 | 상태 확인 | 사유 |
|---|---|---|
| 사업자등록 | 필수 | 본인인증·실결제·알림톡 계약의 공통 선행 조건 |
| 본인인증 계약 (포트원 등) | 필수 | 성인·실명 인증은 번개팅 안전의 근간. 익명 채팅+사진+나이 밴드가 있는 서비스에 미성년자 유입 시 서비스 전체가 위험 |
| 실결제 계약 (토스) | 필수 | 추가 요금 2만원의 실수취, 성비 미달 자동환불의 실작동 |
| 알림 실채널 (SMS) | 필수 | 확정/취소/우선결제 알림이 console.log면 운영 불가 |
| 이미지 스토리지 (R2 등) | 필수 | 프로필 사진 업로드 (현재 스토리지 미작동) |
| 운영 인력 | 권장 | 채팅 신고 처리·강퇴는 상시 모더레이션 부담 — 1인 운영 + 수능 병행 가능 여부 사전 판단 |
